# Технічне завдання (ТЗ) — платформа «Поступ»

> Документ для розробника/студії, щоб оцінити обсяг і вартість робіт.
> Базується на робочому прототипі: https://ianitskyi.github.io/postup/
> Останнє оновлення: 14.07.2026 · Версія ТЗ: 1.0

## 0. Чи можна зробити це без програміста?

Коротка відповідь: **на етапі перевірки ідеї — так, для справжнього продукту — ні.**

**Що реально зробити без розробника (етап валідації, майже безкоштовно):**
- Лендинг на конструкторі (Tilda / Carrd / Webflow) або наш готовий статичний прототип.
- Прийом заявок вступників через Google Forms.
- Облік кандидатів і донорів у Airtable / Google Sheets.
- Перший «збір» вручну: банка monobank + звіт у соцмережах + переказ вишу за реквізитами.
- Це дозволяє перевірити попит і зібрати перших реальних учасників **без жодного рядка коду**.

**Чому для продукту потрібен програміст (не робіть це на no-code):**
- **Гроші.** Рекурентні (щомісячні) платежі, точний облік на цільовому рахунку, виплати вишам, повернення зі збереженням пріоритетів донора — помилка тут = фінансова й репутаційна катастрофа.
- **Персональні дані.** Зберігання документів і даних неповнолітніх з шифруванням і розмежуванням доступу — юридична відповідальність, no-code платформи не дають потрібного контролю.
- **Інтеграції.** ЄДЕБО, платіжний провайдер, банк-партнер — потребують серверної логіки.
- **Складні ролі й воркфлоу.** Модерація, верифікація, кабінети чотирьох типів користувачів.

**Висновок:** спочатку валідуємо ідею no-code-інструментами (Фаза 0 у [плані дій](ACTION-PLAN.md)), паралельно шукаємо грант, і лише потім замовляємо MVP у розробника за цим ТЗ.

## 1. Мета

Веб-платформа, що з'єднує благодійників із вступниками в українські ЗВО та забезпечує цільове фінансування навчання: збір коштів на цільовий (ескроу) рахунок і виплата **напряму університету** після зарахування.

## 2. Ролі користувачів

| Роль | Можливості |
|---|---|
| **Гість** | Перегляд каталогу, профілів, стипендій; разова пожертва без реєстрації |
| **Благодійник** | Кабінет: історія пожертв, підписки, пріоритети, іменні стипендії, звіти |
| **Вступник** | Створення/редагування профілю, завантаження документів і відео, оновлення таймлайну вступу, звіти про навчання |
| **Адміністратор** | Модерація заявок, верифікація, керування зборами, донатами, виплатами, стипендіями |
| **Університет-партнер** | Кабінет: підтвердження зарахувань і успішності, реквізити, перегляд надходжень |

## 3. Функціональні модулі

### 3.1. Профілі вступників
- Реєстрація, майстер створення профілю (кілька кроків).
- Поля: особисті дані, спеціальність, виш, категорії, ціль збору, мотиваційний лист, досягнення, рекомендації.
- Завантаження медіа: фото, відеовізитка, скани документів (приватні).
- Статуси вступу + публічний таймлайн.
- Згода на обробку/публікацію даних (для неповнолітніх — згода батьків), можливість відкликати.

### 3.2. Каталог і пошук
- Список верифікованих профілів, картки з прогресом збору.
- Фільтри: галузь, університет, соціальний критерій; текстовий пошук.
- Фільтр вишів показує лише активні збори.

### 3.3. Пожертви
- Разова, підписка (рекурентна), добровільний внесок «на платформу» (%).
- **Пріоритети донора** на випадок, якщо кошти не знадобляться (невступ / недозбір / бюджет / інший грант): стипендія цьому студенту → інший вступник → повернення 100%.
- Інтеграція платіжного провайдера (LiqPay / WayForPay / Fondy) з підтримкою рекурентних платежів.
- Анонімні пожертви (з урахуванням фінмоніторингу для великих сум).

### 3.4. Цільовий рахунок (ескроу) та облік
- Внутрішній реєстр (ledger): кожен збір — окремий цільовий облік.
- Кошти «заморожені» до настання умови (наказ про зарахування).
- Логіка розблокування, повернення, переспрямування за пріоритетами.
- Повний аудит-лог усіх операцій.

### 3.5. Виплати університетам
- Формування виплати після підтвердження зарахування.
- Реквізити вишів, призначення платежу, статуси (заплановано/виконано), платіжні документи.
- Двофакторне підтвердження виплати (підпис двох адміністраторів).

### 3.6. Іменні стипендії
- Два типи: разова (на вступ) і регулярна (на час навчання).
- Створення: назва, критерії, розмір, тривалість, спосіб поповнення фонду.
- Підбір отримувачів за критеріями, затвердження засновником, виплати через виш.

### 3.7. Звітність
- Публічний таймлайн вступу.
- Разові донори: фінальний звіт (наказ + підтвердження виплати).
- Підписники: семестрові звіти про успішність.
- Email-сповіщення про ключові події.

### 3.8. Адмін-панель
- Дашборд (KPI, графіки надходжень).
- Модерація заявок із чек-лістом верифікації.
- Керування зборами, донатами, виплатами, стипендіями, партнерами.

### 3.9. Верифікація через ЄДЕБО
- Автоматична звірка поданих заяв і статусів зарахування через відкриті дані ЄДЕБО (vstup.edbo.gov.ua).

### 3.10. Кабінет університету-партнера
- Підтвердження зарахувань і семестрової успішності, керування реквізитами.

## 4. Нефункціональні вимоги
- **Безпека:** шифрування персональних даних і документів, розмежування доступу, захист від OWASP Top-10, зберігання платіжних даних лише на боці провайдера (PCI DSS — не зберігаємо карток).
- **Відповідність:** ЗУ «Про захист персональних даних»; для закордонних донорів — GDPR.
- **Продуктивність:** швидке завантаження, оптимізація медіа.
- **Доступність:** адаптив (моб/десктоп), базова A11y.
- **Локалізація:** українська; англійська — бажано (для діаспорних донорів).
- **Аналітика:** приватна, без рекламних трекерів.

## 5. Рекомендований стек (на розсуд розробника)
- **Фронтенд:** React / Next.js.
- **Бекенд:** Node.js (NestJS) або Python (Django/FastAPI).
- **БД:** PostgreSQL. Файли: захищене об'єктне сховище (S3-сумісне).
- **Хостинг:** Vercel/Netlify (фронт) + керований бекенд/БД (напр. Supabase, Railway, або власний VPS).
- **Платежі:** LiqPay / WayForPay / Fondy (рекурентні платежі).
- **Пошта:** транзакційний сервіс (Postmark / SendGrid / власний SMTP).

## 6. Етапність (для оцінки вартості окремо)
- **MVP (мінімум для пілоту):** профілі + верифікація (ручна), каталог, разові пожертви, ручні виплати, базова адмінка, email. *Без рекурентних платежів і ЄДЕБО-інтеграції.*
- **Фаза 2:** підписки (рекурентні платежі), пріоритети, іменні стипендії, кабінет донора.
- **Фаза 3:** ЄДЕБО-інтеграція, кабінети університетів, розширена звітність, англійська локалізація.

## 7. Що надати розробнику для оцінки
1. Це ТЗ + доступ до прототипу (репозиторій + живе демо) як референс UI/UX.
2. Прохання оцінити **окремо**: MVP, Фазу 2, Фазу 3 — у годинах і грошах.
3. Питання до розробника: рекомендований стек, строки, склад команди, вартість підтримки після запуску, досвід із платіжними інтеграціями в Україні.
